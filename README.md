口腔诊所轻量化管理系统需求说明书
1 引言
1.1 文档目的
本文档为口腔诊所轻量化管理系统的正式需求说明书，明确系统建设目标、业务场景、功能需求、非功能需求、数据规则及运行要求。旨在为研发开发、测试验收、诊所落地使用提供统一、可落地的依据，规避功能冗余、操作复杂等问题，适配中小型口腔诊所极简、高效的客户与诊疗管理需求。
本系统定位轻量级、低成本、易上手、免复杂运维、全终端自适应，聚焦核心业务，舍弃大型口腔管理系统的冗余功能，贴合个体诊所、小型口腔门诊无固定电脑、依赖手机日常维护的运营场景。
1.2 项目背景
当前多数小型口腔诊所依赖纸质台账、Excel表格记录客户信息与诊疗数据，存在数据零散、查询困难、诊疗记录追溯不便、客户回访无依据、数据易丢失等问题。同时缺乏标准化的客户档案管理与诊疗流程沉淀，不利于客户维护、复诊跟进与诊所业务复盘。
为解决以上痛点，建设本轻量化口腔诊所管理系统，聚焦客户档案管理、诊疗记录留存、客户跟进管理三大核心场景，实现诊所业务数字化、简易化管理，提升接诊效率与客户服务质量。
1.3 适用范围
本系统适用于单店小型口腔诊所、个体牙医门诊，核心使用者为诊所医生、前台管理人员。系统覆盖客户信息登记、诊疗过程记录、客户回访跟进、基础数据查询统计全流程，不涉及连锁分店管理、复杂财务结算、大型设备对接等重型功能。
1.4 术语与缩略语
•轻量化系统：功能聚焦核心业务，操作极简，部署简单，无需专业运维，硬件与服务器成本低的管理系统
•客户档案：诊所就诊患者的基础个人信息、口腔基础情况、就诊历史汇总档案
•诊疗记录：患者单次就诊的检查结果、诊断结论、治疗项目、用药情况、医嘱、收费、复诊建议等完整记录
•回访跟进：针对术后、治疗后患者的定期随访、复诊提醒、口腔护理叮嘱管理
2 总体概述
2.1 系统建设目标
打造一套极简、高效、易用、稳定的口腔诊所专属管理系统，替代传统纸质与表格管理方式，实现三大核心目标：
1.客户数字化建档：统一存储患者基础信息、口腔病史、过敏史等资料，实现客户信息永久留存、快速查询；
2.诊疗流程标准化留存：每一次就诊、治疗、手术记录可追溯，规范诊疗数据，辅助医生复盘与后续治疗；
3.精细化客户维护：支持复诊提醒、术后回访、客户标签管理，提升客户粘性与复诊率；
4.轻量化低成本运营：无需专业IT运维，界面简洁、操作零基础上手，适配小型诊所人员操作能力。
2.2 系统核心定位
核心定位：小而精的口腔客户&诊疗台账管理系统
摒弃大型HIS系统冗余功能，只保留诊所刚需功能，优先保障数据录入便捷、查询快速、记录完整，满足90%小型口腔诊所日常管理需求。
2.3 用户角色说明
系统设置两类基础角色，权限清晰、极简管控：
•管理员（诊所负责人/前台）：拥有全部权限，可新增、编辑、删除客户档案与诊疗记录，管理用户账号，查看全部数据；
•医生（普通操作员）：可新增、查看、编辑本人接诊的客户档案与诊疗记录，仅可查看数据，无账号管理、数据删除权限。
3 功能性需求
本系统功能模块分为：系统管理模块、客户档案模块、诊疗记录模块、回访跟进模块、基础数据统计模块，所有功能遵循“操作最少步骤、字段最少冗余”原则设计。
3.1 系统管理模块
3.1.1 账号登录与权限管理
•支持账号密码登录，区分管理员、医生两种角色权限；
•管理员可新增、禁用、修改操作员账号，重置登录密码；
•支持登录状态留存，退出登录自动清空会话，保障数据安全；
•权限隔离：医生仅可操作本人接诊数据，管理员可操作全量数据。
3.1.2 系统基础设置
•支持诊所基础信息配置（诊所名称、联系方式、地址，可展示在记录表单底部）；
•支持基础数据字典自定义：治疗项目、口腔病症、常用药品、回访周期等，可自行新增、修改、删除，适配诊所个性化业务；
•支持数据手动备份、本地导出，防止数据丢失。
3.2 客户档案管理模块（核心）
用于统一管理所有就诊患者的基础信息与口腔基础档案，实现一户一档、永久留存。
3.2.1 客户新增建档
支持手动新增客户档案，核心录入字段（无冗余必填项）：
•基础信息：姓名、性别、年龄、联系电话、微信号、居住地址；
•口腔基础信息：既往口腔病史、药物过敏史、全身基础病史（高血压、糖尿病等）、牙齿基础情况；
•客户标签：支持自定义标签（新客户、老客户、高需求客户、正畸客户、补牙客户等），便于分类管理；
•建档时间、首次接诊医生自动记录，无需手动输入。
3.2.2 客户档案编辑与查询
•支持通过姓名、电话、标签快速检索客户，一键调取完整档案；
•支持随时编辑客户信息，修改记录自动留存更新时间；
•支持客户档案一键导出（Excel格式），便于线下备份；
•支持客户信息禁用/删除（仅管理员权限），避免无效数据堆积。
3.2.3 客户档案汇总
单个客户档案页面可汇总展示该患者全部历史诊疗记录、回访记录、消费记录，实现单点查看客户全生命周期就诊数据。
3.3 诊疗记录管理模块（核心）
记录患者每一次就诊的完整诊疗信息，是系统核心业务模块，实现诊疗过程可追溯、医嘱可留存。
3.3.1 新增诊疗记录
选择对应客户后，新增单次就诊记录，核心录入字段：
•基础就诊信息：就诊时间、接诊医生、就诊类型（初诊/复诊/急诊）；
•检查诊断信息：主诉、口腔检查结果、拍片检查结论、确诊病症；
•治疗信息：治疗项目（支持多选）、治疗部位、治疗过程简述；
•用药信息：药品名称、用量、用法、服用周期；
•费用信息：本次诊疗总费用、已付金额、未付余额（简易记账，无需复杂财务）；
•医嘱与复诊：术后护理叮嘱、禁忌事项、建议复诊时间、下次检查项目；
•附件上传：支持上传口腔拍片、牙齿照片、检查报告等图片文件，绑定对应诊疗记录。
3.3.2 诊疗记录查询与编辑
•支持按客户姓名、就诊时间、接诊医生、治疗项目筛选查询诊疗记录；
•支持对未归档的诊疗记录进行编辑修改，已完成记录仅可查看、补充，不可随意篡改核心诊断内容；
•单条诊疗记录支持打印、导出，可作为就诊凭证留存。
3.3.3 历史诊疗追溯
自动汇总同一客户所有历史诊疗记录，按时间倒序排列，医生可快速查看患者过往治疗情况，为本次诊疗提供参考，避免重复治疗、用药冲突。
3.4 客户回访跟进模块
针对术后、治疗后、待复诊客户进行跟进管理，提升客户服务与复诊率。
3.4.1 回访计划创建
医生完成诊疗后，可直接创建回访任务：设置回访时间、回访类型（术后回访、复诊提醒、常规问候）、回访重点内容。
3.4.2 回访记录登记
完成电话/微信回访后，登记回访结果：客户反馈、恢复情况、是否预约复诊、后续跟进需求。
3.4.3 待回访提醒
系统首页展示当日待回访、待复诊客户清单，直观提醒工作人员跟进，避免遗漏客户维护。
3.5 基础数据统计模块
轻量化统计，无需复杂报表，满足诊所日常复盘需求：
•时间段就诊统计：每日/每月接诊人数、初诊/复诊人数统计；
•治疗项目统计：指定时间段内各类治疗项目（补牙、正畸、拔牙等）开展次数；
•客户统计：新增客户数量、老客户复诊数量统计；
•所有统计数据支持可视化简易图表展示、支持Excel导出。
4 非功能性需求
4.1 易用性需求
•界面简洁清晰，功能分区明确，符合诊所工作人员操作习惯，零基础人员10分钟可上手操作；
•核心操作（建档、新增诊疗记录）步骤不超过3步，减少繁琐录入；
•必填字段最小化，非核心信息可选录入，针对手机端录入场景优化表单布局，减少移动端输入繁琐问题，提升移动运维效率。
4.2 性能需求
•系统响应速度快，页面加载、数据查询响应时间≤1秒；
•支持多终端、多用户同时在线操作，电脑端、手机端可同步登录使用，数据实时同步、无卡顿、无数据延迟；
•支持多终端同时在线操作（前台、医生同时登录使用无卡顿）。
4.3 安全性需求
•账号密码加密存储，支持密码修改、找回；
•角色权限隔离，防止误操作删除、篡改核心诊疗数据；
•支持数据手动备份与定期自动备份，防止数据丢失；
•患者隐私数据加密展示，禁止非授权人员查看客户隐私信息。
4.4 兼容性需求
•全终端自适应适配：系统采用响应式布局，完美适配电脑端、手机端、平板端，无需单独开发移动端小程序/APP，浏览器直接访问即可自动适配对应屏幕尺寸，满足诊所无电脑、手机日常运维的核心场景；
•电脑端：适配主流Chrome、Edge、360浏览器，兼容常规办公电脑屏幕分辨率，满足精细化录入、数据查看、打印导出需求；
•手机/平板端：适配安卓、iOS移动端设备，页面自动缩放、按钮自适应放大，适配手机单手、双手操作，优化移动端录入表单、查询、回访登记等核心操作体验；
•支持Windows、安卓、iOS系统运行，无需专属服务器，普通办公电脑、手机均可随时随地登录使用。
4.5 可维护性需求
•系统轻量化部署，无需专业IT运维人员维护；
•功能模块化设计，后续可按需新增简易功能（如简易预约、耗材登记），支持迭代升级；
•数据结构清晰，备份、导出、迁移操作简单。
5 运行环境需求
5.1 硬件环境（低配即可满足）
•客户端：普通办公电脑、笔记本、智能手机、平板，无特殊硬件配置要求；
•服务端：支持本地部署或轻量云部署，无需独立高端服务器。
5.2 软件环境
•操作系统：Windows 10/11、安卓、iOS全平台兼容；
•运行环境：适配电脑、手机、平板主流浏览器，无需安装客户端，网页端直接登录使用，支持随时随地移动端维护信息；
•数据存储：轻量化数据库，支持本地存储。
6 业务规则与数据约束
1.客户唯一标识：以客户手机号为唯一检索标识，避免重复建档；
2.诊疗数据溯源：所有诊疗记录自动绑定接诊医生与时间，操作日志可追溯；
3.数据防误删：核心诊疗记录删除需管理员二次确认，普通医生无删除权限；
4.隐私保护：客户电话、住址等隐私信息仅内部账号可查看，无公开导出权限；
5.表单规范：必填字段为空时无法提交，保证核心数据完整。
